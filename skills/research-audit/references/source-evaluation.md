# Source Evaluation

Use this reference when a research task depends on third-party claims, public repositories, product pages, documentation, papers, pricing, regulations, or fast-moving facts.

## Source priority

1. Primary sources: official docs, repository files, release notes, standards, papers, filings, or direct product pages.
2. Maintainer-authored sources: issue comments, PRs, discussions, changelogs, or project roadmaps from maintainers.
3. Reputable secondary analysis: established publications, expert writeups, or benchmark reports.
4. Community signals: forums, social posts, examples, and anecdotal usage reports.

## Evaluation checks

- Record publication or commit dates when recency matters.
- Separate observed repository facts from inferred design intent.
- Compare adoption value against maintenance cost and project fit.
- Check license and copying risk before recommending code reuse.
- Prefer concept transfer over direct code adoption unless explicitly approved.

## Output expectations

A research recommendation should state what was verified, what remains uncertain, what should be adopted, what should be avoided, and the safest implementation sequence.
